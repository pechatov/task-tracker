# MVP Development Plan

Этот документ фиксирует рабочий план MVP. Доменный словарь живет в `CONTEXT.md`, архитектурные решения - в `docs/adr/`.

## Product Scope

- Основной экран: Today dashboard.
- Задачи создаются и редактируются вручную.
- Задачи одноразовые; рутины и повторения остаются за пределами MVP.
- Заметки остаются за пределами MVP, но задача может иметь опциональное описание.
- Интеграции календарей в MVP работают только на чтение.

## Navigation

- Основные разделы MVP: Today, Calendar, Projects/Streams, Settings.
- На desktop используется левый sidebar.
- На mobile используется нижняя навигация или компактное меню.
- Today открывается как основной раздел после входа.

## Today Dashboard

- Полноценная форма создания задачи со всеми MVP-полями.
- Блок задач дня: все открытые задачи на сегодня, отсортированные по приоритету дня.
- Блок событий дня: встречи и задачи с временными блоками в хронологическом порядке.
- Блок просроченных задач с быстрым переносом на другую дату.

## Calendar Views

- В MVP входят day и week представления.
- Month представление остается за пределами MVP.
- Date-only задачи можно перетаскивать между днями.
- Задачи с временным блоком можно перетаскивать между временными слотами в day/week представлениях.
- Read-only календарные события из внешних календарей не перетаскиваются.

## Task Editing

- Клик по задаче из Today dashboard или календаря открывает редактирование задачи.
- На desktop редактирование открывается в Notion-style модальном окне поверх затемненного фона.
- На mobile редактирование открывается как полноэкранный экран или fullscreen modal.
- Модальное окно содержит те же поля задачи, что и форма создания.
- Side panel остается запасным вариантом, если модальное окно покажет проблемы с доступностью или адаптивностью.
- Удаление задачи входит в MVP как второстепенное destructive action в редактировании задачи.
- Удаление требует подтверждения; основной workflow остается через состояния выполнена и отменена.

## Reminders

- В MVP входят in-app напоминания при открытом приложении.
- Минимальный сценарий: desktop-приложение показывает заметное уведомление о предстоящей встрече.
- Напоминание показывается за 10 минут до начала события.
- System push notifications, service worker push и VAPID остаются за пределами MVP.
- Напоминание может открывать календарное событие или ссылку события, если она доступна.
- В MVP у напоминания достаточно действия "Закрыть"; snooze не входит в MVP.
- Закрытое напоминание не показывается повторно в текущей сессии приложения.

## Task Fields

- Название.
- Описание задачи, опционально.
- Дата выполнения.
- Приоритет дня.
- Состояние: открыта, выполнена, отменена.
- Стрим, опционально; у стрима есть цвет лейбла и состояние.
- Проект, опционально; при выборе проекта стрим подставляется из проекта; у проекта есть цвет лейбла и состояние.
- Временной блок, опционально.
- В форме задачи можно inline создать новый стрим или проект.

## Projects And Streams

- Раздел Projects/Streams нужен для полноценного управления стримами и проектами.
- У стримов и проектов есть состояние: активен или завершен.
- Проект может завершаться независимо от стрима.
- Если стрим завершен, в форме задачи скрывается сам стрим и все проекты внутри него.
- В форме задачи по умолчанию доступны только активные стримы и активные проекты внутри активных стримов.
- Завершенные стримы и проекты сохраняются на старых задачах и доступны в разделе Projects/Streams.
- Inline creation из формы задачи обязателен для стримов и проектов.
- Inline создание стрима включает минимум имя и цвет.
- Inline создание проекта включает минимум имя, цвет и стрим, к которому относится проект.

## Visual Display

- Цвет стрима или проекта применяется к соответствующему лейблу задачи.
- Цвет не применяется как заливка всей задачи.
- Если у задачи есть проект, показываем проектный лейбл; стрим может отображаться вторичным лейблом или выводиться в деталях задачи.
- У подключенного календаря есть цвет; события календаря показывают его через компактный маркер, например левую полоску или точку.
- Для внешних событий показываем подпись календаря или источника, чтобы отличать их от пользовательских задач.

## Calendar Sync

- Источники MVP: Exchange через EWS (логин и пароль аккаунта), Google Календарь через OAuth и Яндекс.Календарь через CalDAV.
- После подключения источника пользователь выбирает конкретные календари для синхронизации.
- Основной календарь источника включается по умолчанию, остальные календари доступны через toggles в Settings.
- Синхронизация выполняется через scheduled polling в worker-процессе.
- В настройках нужна ручная команда resync.
- Горизонт синхронизации и хранения событий: 365 дней назад и 90 дней вперед от текущей даты.
- Для календарных событий храним минимальные данные для планирования: заголовок, начало, конец, all-day, место, организатор, краткую информацию об участниках, источник, внешний id, время обновления и ссылку события.
- Описание/тело календарного события в MVP не сохраняем.
- По клику на событие пользователь может открыть ссылку события в браузере, если источник ее предоставляет или если URL найден в месте события.
- При извлечении ссылки события сначала используем явный join/event URL от провайдера, затем URL из поля места события.
- При отключении календарного источника удаляем локальные credentials, sync state и кэш событий этого источника.
- Пользовательские задачи при отключении календарного источника не удаляются.
- Webhooks не входят в MVP.

## Access And Security

- MVP работает в single-user режиме.
- Данные сразу scoped by user, чтобы позже добавить нескольких пользователей.
- Вход по email/password.
- Публичной регистрации нет.
- Первый пользователь создается CLI-командой с интерактивным вводом пароля.
- Пароли хранятся только как password hash.
- Секреты календарных интеграций хранятся на сервере в зашифрованном виде.

## Stack

- Next.js, React, TypeScript.
- Postgres.
- Drizzle ORM.
- Отдельный worker для синхронизации календарей и фоновых задач.
- Docker Compose для локального запуска и self-hosted деплоя.
